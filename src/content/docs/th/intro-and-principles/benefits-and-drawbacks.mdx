## Context

คุณเข้าใจแล้วว่า microservices architecture คืออะไรและมันต่างจาก monolith อย่างไร ก่อนตัดสินใจว่าจะรับมันมาใช้หรือไม่ คุณต้องมีบัญชีที่ซื่อตรง: สิ่งที่สถาปัตยกรรมนี้ให้คุณจริง ๆ และสิ่งที่มันเรียกเก็บจากคุณจริง ๆ การตลาดและทอล์กในงาน conference มักจะไล่ลิสต์ประโยชน์แล้วข้ามใบเรียกเก็บเงินไป บทเรียนนี้วางทั้งสองคอลัมน์ไว้เคียงข้างกัน เพราะคุณจะได้ประโยชน์ก็ต่อเมื่อ *จ่าย* ต้นทุน — ทั้งสองเป็นสองด้านของการตัดสินใจเดียวกัน

## Problem

อันตรายคือการรับ microservices มาใช้เพื่อข้อดี ในขณะที่ประเมินข้อเสียต่ำเกินไป ทีมได้ยินคำว่า "scale อย่างอิสระ" และ "deploy ได้เร็วขึ้น" แล้วก็ขยับตัวตาม เพียงเพื่อจะค้นพบว่าตอนนี้ request หนึ่งวิ่งข้ามถึงห้า service, transaction เดียวกลายเป็น workflow หลายขั้นตอน และ bug ง่าย ๆ ตอนนี้ต้องอาศัยการเชื่อมโยง log จากสาม process

ดังนั้นคำถามที่บทเรียนนี้ตอบคือ: สำหรับแต่ละประโยชน์ที่ microservices สัญญาไว้ ต้นทุนที่คู่กันคืออะไร และคุณจะชั่งน้ำหนักทั้งสองอย่างไร?

## Solution

วาง trade-off ออกมาเป็นสองคอลัมน์ที่เชื่อมโยงกัน ทุกประโยชน์ทางซ้ายมีต้นทุนที่สอดคล้องอยู่ทางขวา — พวกมันมาด้วยกัน ไม่ได้มาแยกกัน

### ประโยชน์

- **การ deploy และ scale อย่างอิสระ** แต่ละ service ship ตามตารางของตัวเอง ดังนั้นการเปลี่ยนแปลงเล็ก ๆ ใน service หนึ่งไม่จำเป็นต้อง deploy ใหม่ทั้งระบบ คุณยังสามารถรัน instance ของเฉพาะ service ที่ต้องการเพิ่มได้ แทนที่จะ scale ทั้งแอปพลิเคชัน
- **ความเป็นอิสระของทีม** ทีมหนึ่งสามารถเป็นเจ้าของ service ตั้งแต่ต้นจนจบ — โค้ดของมัน ข้อมูลของมัน release pipeline ของมัน — และตัดสินใจในระดับท้องถิ่นได้โดยไม่ต้องประสานงาน codebase ที่ใช้ร่วมกันหรือ deploy ที่ใช้ร่วมกันกับคนอื่นทุกคน
- **fault isolation** ความล้มเหลวสามารถถูกจำกัดให้อยู่ใน service เดียวได้ ถ้า recommendations service ล้มลง checkout service ก็ยังรับ order ต่อได้ ตราบใดที่คุณออกแบบขอบเขตให้เสื่อมสภาพอย่างนุ่มนวล (degrade gracefully)
- **ความหลากหลายทางเทคโนโลยี** แต่ละ service สามารถเลือกภาษา framework หรือ datastore ที่เหมาะกับงานของมัน และคุณสามารถรับเทคโนโลยีใหม่มาใช้ใน service เดียวได้โดยไม่ต้องเขียนทุกอย่างใหม่

### ข้อเสีย

- **ความซับซ้อนของระบบแบบกระจาย** ตอนนี้เครือข่ายอยู่ตรงกลางแอปพลิเคชันของคุณ การเรียกอาจช้า ล้มเหลว timeout หรือมาถึงสองครั้ง พฤติกรรมที่เคยเป็น function call แบบ deterministic กลายเป็น request ที่ต้องรับมือกับ partial failure
- **ความสอดคล้องของข้อมูลยากขึ้น** เมื่อแต่ละ service เป็นเจ้าของข้อมูลของมัน คุณสูญเสีย atomic transaction เดียวไป การรักษาความสอดคล้องของข้อมูลข้าม service หมายถึง eventual consistency, saga และ compensating action — มีทั้งโมดูลของคอร์สนี้ที่มีอยู่ก็เพราะต้นทุนข้อนี้ข้อเดียว
- **ภาระการดำเนินงาน** ตอนนี้คุณ deploy, monitor และรักษาความปลอดภัยให้ service จำนวนมากแทนที่จะเป็นตัวเดียว สิ่งนี้เรียกร้องการลงทุนจริงจังใน automation, CI/CD, container orchestration และ logging กับ tracing แบบรวมศูนย์ ก่อนที่สถาปัตยกรรมจะคุ้มทุน
- **การทดสอบข้าม service** unit test ไม่เปลี่ยน แต่การตรวจสอบว่า service ทำงาน *ร่วมกัน* ได้ตอนนี้เกี่ยวข้องกับเครือข่าย contract ระหว่าง service และ failure mode ของการเรียกแบบกระจาย end-to-end test ยากขึ้นอย่างมีนัยสำคัญ

## Resulting context

บทสรุปที่ซื่อตรงคือ microservices ย้ายความซับซ้อน มันไม่ได้กำจัดมัน ใน monolith ความซับซ้อนอยู่ภายในโค้ด ที่ซึ่ง compiler และ debugger ช่วยคุณได้ ใน microservices ส่วนใหญ่ของมันย้ายไปอยู่ในพื้นที่ *ระหว่าง* service — เครือข่าย ข้อมูล การดำเนินงาน — ที่ซึ่งเครื่องมือของคุณอ่อนแอกว่าและ failure mode ของคุณยุ่งเหยิงกว่า

การแลกเปลี่ยนนั้นคุ้มที่จะทำเมื่อประโยชน์พุ่งเป้าไปยังปัญหาที่คุณมีจริง: ทีมที่ติดขัดเพราะกันและกัน component ที่มีความต้องการ scale ต่างกันสุดขั้ว หรือข้อกำหนดเรื่อง fault isolation ที่คุณไม่สามารถทำได้ใน process เดียว มันเป็นการแลกเปลี่ยนที่แย่เมื่อคุณรับต้นทุนมาเพื่อไล่ตามประโยชน์ที่คุณยังไม่ต้องการ บทเรียนถัดไปจะแปลงการพิจารณานี้ให้เป็นการตัดสินใจที่เป็นรูปธรรม
