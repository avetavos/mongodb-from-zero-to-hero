## Context

คุณสามารถนิยาม microservices ได้ และท่อง trade-off ของมันได้ ตอนนี้มาถึงการตัดสินใจที่สำคัญจริง ๆ บนโปรเจกต์จริง: *ระบบนี้ควรเป็น microservices architecture หรือไม่ และถ้าควร เมื่อไร?* นี่คือจุดที่วิจารณญาณทางวิศวกรรมที่ดีจะแสดงออกมา เพราะคำตอบที่ผิดในทิศทางใดก็ตามมีราคาแพง decompose เร็วเกินไปแล้วคุณจะจ่ายต้นทุนของระบบแบบกระจายก่อนที่จะมี scale มากพอจะให้เหตุผลได้ decompose ช้าเกินไปแล้วคุณจะต้องสู้กับ monolith ที่กลายเป็นคอขวดสำหรับทีมขนาดใหญ่

## Problem

ไม่มีกฎสากล แต่มีสัญญาณ ความผิดพลาดคือการตัดสินใจตามแฟชั่น ("ตอนนี้ใคร ๆ ก็ใช้ microservices") หรือความกลัว ("monolith ของเรารู้สึกยุ่งเหยิง มันต้องผิดแน่ ๆ") monolith ที่ยุ่งเหยิงมักเป็นสัญญาณว่าคุณต้องการขอบเขตภายในที่สะอาดกว่า ไม่จำเป็นต้องเป็นเครือข่ายระหว่าง module ของคุณ

ดังนั้นคำถามคือ: ปัจจัยที่เป็นรูปธรรมข้อไหนควรผลักคุณไปสู่ microservices และข้อไหนควรรั้งคุณไว้กับ monolith?

## Solution

ชั่งน้ำหนักสามปัจจัยไปด้วยกัน — พวกมันเสริมกันและกัน และไม่มีปัจจัยใดปัจจัยเดียวที่ชี้ขาดได้ด้วยตัวเอง:

- **ขนาดทีม** นักพัฒนาเพียงหยิบมือ ship ได้เร็วที่สุดบน codebase เดียว เมื่อคุณมีหลายทีมแย่งชิง codebase เดียวและ deploy pipeline เดียว ต้นทุนการประสานงานก็พุ่งขึ้น และการให้แต่ละทีมมี service ที่ deploy ได้ของตัวเองสามารถลบการแย่งชิงนั้นออกไป
- **ความซับซ้อนของ domain** domain เล็ก ๆ ที่เข้าใจกันดีอยู่แล้วแทบไม่ต้องการการแยก domain ขนาดใหญ่ที่มีพื้นที่ย่อยชัดเจนซึ่งวิวัฒน์ในอัตราที่ต่างกันและมีขอบเขตตามธรรมชาติ คือผู้สมัครสำหรับการ decompose ไปตามขอบเขตเหล่านั้น
- **ความต้องการ scale** ถ้าทั้งระบบ scale ได้ดีในฐานะหน่วยเดียว คุณก็ไม่จำเป็นต้องแยกมันเพื่อ scale ถ้า component เฉพาะมีโปรไฟล์ load ที่ต่างกันมาก — ตัวหนึ่ง CPU-bound อีกตัว memory-bound อีกตัวได้ traffic มากกว่าสิบเท่า — การ scale อย่างอิสระก็กลายเป็นเหตุผลจริงในการแยกพวกมัน

### Monolith first

มีค่าเริ่มต้นที่หนักแน่นอยู่ตรงนี้ มักเรียกกันว่า **monolith first**: เริ่มด้วย monolith ที่มีโครงสร้างดี ทำให้ผลิตภัณฑ์ถูกต้อง และเรียนรู้ว่าขอบเขตที่แท้จริงใน domain ของคุณอยู่ตรงไหน ขอบเขตของผลิตภัณฑ์ที่ยังอ่อนวัยเป็นเพียงการเดา และการเดามันผิดนั้นแก้ไขในราคาที่ถูกกว่ามากภายใน codebase เดียว เทียบกับการแก้ข้ามเครือข่ายของ service เมื่อระบบเติบโตและคุณสังเกตเห็นความเจ็บปวดที่แท้จริง — ทีมที่ติดอยู่กับ deploy ที่ใช้ร่วมกัน component ที่ต้อง scale ด้วยตัวเอง — คุณก็ดึงชิ้นส่วนนั้นออกมาเป็น service คุณ decompose ไปตามขอบเขตที่คุณ *ค้นพบ* ไม่ใช่ขอบเขตที่คุณ *จินตนาการ* ขึ้นในวันแรก

นี่ไม่ได้หมายความว่า monolith แบบลวก ๆ ทรัพย์สินที่ทำให้การดึงออกในภายหลังราคาถูกคือ **modular** monolith: monolith ที่มีขอบเขต module ภายในที่สะอาด ทำให้การดึง module ออกมาเป็น service เป็น mechanical refactor มากกว่าการแกะปมที่พันกันยุ่ง

## Resulting context

เมื่อสามปัจจัยเรียงตัวกัน — องค์กรขนาดใหญ่ domain ที่ซับซ้อนซึ่งมีรอยต่อตามธรรมชาติ และ component ที่มีความต้องการ scale แตกต่างกัน — *และ* คุณมีวุฒิภาวะด้านการดำเนินงานพอที่จะรัน service จำนวนมาก microservices ก็มีแนวโน้มจะคุ้มทุน เมื่อพวกมันไม่เรียงตัวกัน modular monolith แทบจะเป็นตัวเลือกที่ดีกว่า ถูกกว่า เร็วกว่าเสมอ และมันก็เปิดทางเลือกในการ decompose ภายหลังไว้

ความจริงที่ช่วยให้อุ่นใจคือ นี่ไม่ค่อยเป็นการตัดสินใจที่ย้อนกลับไม่ได้แบบทั้งหมดหรือไม่เลย คุณเริ่มต้นในฐานะ monolith ได้ และค่อย ๆ ลอก service ออกมาทีละตัวเมื่อความจำเป็นกลายเป็นจริง — ดึง component ที่เจ็บที่สุดออกก่อน และทิ้งส่วนที่เหลือไว้ใน monolith ตราบเท่าที่มันยังรับใช้คุณได้
