## บริบท

ในโลกของ monolith การดีบักแทบจะเป็นเรื่องที่จับต้องได้ คำขอเข้ามา วิ่งผ่านโปรเซสเดียว แตะ log file ไฟล์เดียว แล้วก็คืนค่ากลับมาหรือไม่ก็โยน stack trace ที่คุณสามารถอ่านได้ตั้งแต่ต้นจนจบ เมื่อมีอะไรผิดพลาด คุณก็เปิด terminal, tail `app.log`, แนบ profiler และเฝ้าดูเรื่องราวทั้งหมดคลี่คลายในที่เดียว

จากนั้นคุณก็แยกโปรเซสนั้นออกเป็นยี่สิบบริการที่รันบน replica หลายสิบตัวอยู่หลัง load balancer คลิกเดียวของผู้ใช้ตอนนี้แตกออกเป็นต้นไม้ของการเรียกที่กระจายไปตามเครื่องที่คุณอาจไม่เคยล็อกอินเข้าไปเลย, replica ที่เกิดและดับไปทุกครั้งที่ deploy และโปรเซสที่ local log file หายวับไปในวินาทีที่ container ถูกรีไซเคิล

## ปัญหา

นิสัยเดิม ๆ ที่ว่า SSH เข้าเครื่องแล้วมองหาดู เงียบ ๆ ก็ใช้ไม่ได้แล้ว เครื่องไหนล่ะ คำขอนั้นแตะมันไปแปดเครื่อง replica ที่ให้บริการมันถูก autoscaler ฆ่าทิ้งไปเมื่อสองนาทีที่แล้ว พร้อมพา log ของมันไปด้วย ข้อผิดพลาดที่ลูกค้าของคุณรายงานเป็น HTTP 500 ที่ gateway คืนกลับมา ซึ่งเบาะแสเดียวของมันคือ downstream timeout ที่มันบอกชื่อไม่ได้

ดังนั้นคำถามหลักของโมดูลนี้คือ เมื่อไม่มีเครื่องเดียวที่ถือความจริงทั้งหมด คุณจะเข้าใจได้อย่างไรว่าระบบของคุณกำลังทำอะไรอยู่ ดีพอที่จะดีบักเหตุการณ์ตอนตีสาม และดีพอที่จะสังเกตปัญหาก่อนที่ลูกค้าจะสังเกตเห็น

## ทางออก

คำตอบคือ **observability** การ instrument ทุกบริการเพื่อให้พฤติกรรมของมันสามารถถูกสร้างขึ้นใหม่จากภายนอกได้ จากสัญญาณที่มันปล่อยออกมา โดยไม่ต้องล็อกอินเข้าไปที่ใดเลย สัญญาณเหล่านั้นมาในสี่รูปแบบที่เสริมกัน และคุณต้องการทั้งสี่อย่าง

แต่ละ pattern ในโมดูลนี้ครอบคลุมหนึ่งในสัญญาณเหล่านั้น

- **Health Check API** — ทุกบริการเปิดเผย endpoint ที่รายงานว่ามันยังมีชีวิตอยู่และพร้อมให้บริการหรือไม่ เพื่อให้ orchestrator และ load balancer สามารถ route หลบ instance ที่ป่วยได้โดยอัตโนมัติ
- **Log Aggregation** — ส่ง structured logs จากทุก replica ไปยัง store กลางที่ค้นหาได้ ผูกเข้าด้วยกันด้วย correlation id เพื่อให้คำขอหนึ่งอ่านเป็นเรื่องราวเดียว
- **Distributed Tracing** — ให้ trace id แก่ทุกคำขอจากภายนอก, propagate มันข้ามทุก hop และสร้าง call tree เต็มรูปแบบพร้อม timing ขึ้นใหม่ เพื่อให้คุณเห็นว่า latency และข้อผิดพลาดอยู่ที่ใดจริง ๆ
- **Application Metrics** — เปิดเผย counters, gauges และ histograms สำหรับ request rate, errors และ latency เพื่อให้ dashboards และ alerts สามารถสรุปสุขภาพของทั้ง fleet ได้ในพริบตา
- **Audit Logging** — บันทึกว่าใครทำอะไรและเมื่อใด เป็น record ที่คงทนและตรวจจับการดัดแปลงได้ แยกออกจาก debug logs เพราะมันตอบคำถามที่ต่างออกไป

## โมดูลนี้ครอบคลุมอะไรบ้าง

เราเริ่มจากสัญญาณที่ถูกที่สุดและเชิงปฏิบัติการมากที่สุด (Health Check API), เคลื่อนผ่านสามเสาหลักที่อธิบายพฤติกรรม (logs, traces, metrics) และจบด้วย record เฉพาะทางที่ตอบโจทย์ด้านความปลอดภัยและการกำกับดูแล (Audit Logging) แต่ละบทเรียนมีรูปแบบเดียวกัน คือ สถานการณ์ที่นำไปสู่ pattern, แรงที่ตึงเครียดกัน, สิ่งที่ pattern ทำ, ไดอะแกรม, โค้ด instrumentation ที่รันได้หรือตัวอย่าง log ที่เป็นรูปธรรม และการมองอย่างซื่อตรงว่ามันมีต้นทุนอะไรกับคุณบ้าง

## Pattern ที่เกี่ยวข้อง

- [Health Check API](/microservices-design-from-zero-to-hero/th/observability/health-check-api/) — เริ่มที่นี่ มันเป็นสัญญาณที่ง่ายที่สุดและเป็นสิ่งที่ platform ของคุณพึ่งพาก่อนเป็นอันดับแรก
- [Distributed Tracing](/microservices-design-from-zero-to-hero/th/observability/distributed-tracing/) — pattern ที่เข้ามาแทนที่ "อ่าน stack trace เดียว" ในโลกแบบกระจายได้ตรงที่สุด

<Callout title="Monitoring ไม่ใช่ observability">
Monitoring ตอบคำถามที่คุณรู้อยู่แล้วว่าจะต้องถาม — "CPU เกิน 80 เปอร์เซ็นต์หรือยัง?" Observability คือคุณสมบัติที่ทำให้คุณสามารถถามคำถาม *ใหม่* เกี่ยวกับระบบของคุณได้หลังจากที่มันรันอยู่แล้ว โดยไม่ต้อง ship โค้ดใหม่ สัญญาณทั้งสี่ที่นี่คือวัตถุดิบ เป้าหมายคือการสามารถดีบักความล้มเหลวที่คุณไม่เคยเห็นมาก่อนได้จากภายนอก
</Callout>
