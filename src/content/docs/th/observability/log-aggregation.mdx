## บริบท

ทุกบริการเขียน logs — นิสัยนั้นรอดมาได้อย่างไม่บุบสลายเมื่อย้ายมาสู่ microservices สิ่งที่เปลี่ยนไปคือที่ที่ logs เหล่านั้นอยู่ ใน monolith มีโปรเซสเดียวและ `app.log` เดียว; `grep` บนไฟล์นั้นคือ debugger ของคุณ ตอนนี้คำขอเชิงตรรกะเดียวกันแตะห้าบริการ แต่ละบริการรัน replica หลายตัว แต่ละตัวเขียนลง local file หรือ standard output ของตัวเองภายใน container ที่ scheduler อาจทำลายได้ทุกเมื่อ

ลูกค้ารายงานว่า checkout ของพวกเขาล้มเหลวตอน 14:32 บรรทัดที่เกี่ยวข้องกระจัดกระจายอยู่ตาม order replica ที่จัดการคลิก, payment replica ที่มันเรียก และ inventory replica ที่มันเรียกหลังจากนั้น — สามไฟล์บนสามเครื่อง ซึ่งสองในนั้นถูกรีไซเคิลไปแล้วและหายไป

## ปัญหา

ไม่มีที่เดียวให้มองหา และต่อให้คุณล็อกอินเข้าไปทุกเครื่องได้ logs ก็จะไม่เชื่อมต่อกัน order service log ข้อผิดพลาดหนึ่ง, payment service log timeout หนึ่ง, inventory service ไม่ได้ log อะไรเลย — แต่ไม่มีอะไรในบรรทัดเหล่านั้นบอกคุณว่ามันเป็นของคำขอ *เดียวกัน* คุณถูกทิ้งให้นั่งเทียบเคียงด้วย timestamp และการเดา ข้ามไฟล์ที่อาจ rotate ออกไปจากการมีอยู่แล้ว

แล้วคุณจะทำให้บรรทัด log จากคำขอเดียว ที่กระจายอยู่ตาม replica ที่อายุสั้นจำนวนมาก ทั้ง *ค้นหาเจอได้ในที่เดียว* และ *เชื่อมโยงถึงกัน* ได้อย่างไร

## ทางออก

**Log aggregation** มีสองส่วน และคุณต้องการทั้งสอง

ส่วนแรก **pipeline ส่ง logs ออกจากเครื่อง** แต่ละบริการเขียน logs ของมันลง standard output (หรือ local agent tail มัน) และ collector ส่งต่อทุกบรรทัดไปยัง store กลางที่มี index ที่คุณสามารถค้นหาได้ครอบทุกบริการและทุก replica พร้อมกัน เพราะ logs ออกจาก container ทันที มันจึงอยู่รอดต่อจาก replica ที่ผลิตมันออกมา

ส่วนที่สอง logs ถูกทำให้ **มีโครงสร้างและเชื่อมโยง** แทนที่จะเป็นข้อความรูปแบบอิสระ แต่ละบรรทัดเป็น record ที่อธิบายตัวเองได้ — โดยทั่วไปคือ JSON — ที่มี field ที่สอดคล้องกัน คือ timestamp, level, ชื่อบริการ และที่สำคัญคือ **correlation id** (บางครั้งเรียก request id) ที่ถูกสร้างขึ้นครั้งเดียวที่ขอบของระบบ และแนบไปกับทุกบรรทัดที่ปล่อยออกมาในขณะจัดการคำขอนั้น ในทุกบริการที่มันแตะ การค้นหา id เดียวนั้นจะดึงทุกบรรทัดของการเดินทางของคำขอกลับมาตามลำดับ ไม่ว่า replica หรือบริการใดเขียนมัน

## ตัวอย่าง

บรรทัด structured log ที่ payment service ปล่อยออกมาในขณะจัดการ checkout มันเป็น JSON object เดียวต่อบรรทัด ที่มีชื่อ field คงที่และ correlation id ที่ผูกมันเข้ากับทุกบรรทัดอื่นของคำขอเดียวกัน:

```json
{
  "timestamp": "2026-06-25T14:32:07.512Z",
  "level": "error",
  "service": "payment-service",
  "instance": "payment-7d9f-bk2qx",
  "correlationId": "c1f4e9a2-8b3d-4e77-9a01-2f6c5d8e1b04",
  "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",
  "event": "charge_failed",
  "orderId": "ord_88213",
  "amount": 4999,
  "currency": "USD",
  "reason": "gateway_timeout",
  "durationMs": 3021,
  "message": "payment gateway did not respond within 3s"
}
```

การค้นหา store กลางด้วย `correlationId = "c1f4e9a2-8b3d-4e77-9a01-2f6c5d8e1b04"` จะคืนค่าบรรทัดนี้พร้อมกับบรรทัดของ order service ที่เริ่มคำขอ และบรรทัดของ inventory service ที่ตามมา — เรื่องราวเต็มของ checkout เดียว ประกอบขึ้นใหม่จากสามบริการ

## บริบทที่ได้ผลลัพธ์

สิ่งที่คุณได้รับ:

- **ที่เดียวให้ค้นหา ครอบทุกอย่าง** คุณ query ทุกบริการและทุก replica พร้อมกัน และ logs อยู่รอดได้นานกว่า container ที่เขียนมัน ดังนั้น pod ที่ถูกรีไซเคิลจึงไม่พาหลักฐานของคุณไปด้วยอีกต่อไป
- **คำขออ่านเป็นเรื่องราว** การ filter ด้วย correlation id สร้างเส้นทางของคำขอเดียวข้ามขอบเขตบริการขึ้นใหม่ตามลำดับ — กู้คืนมุมมอง end-to-end ที่คุณได้มาฟรี ๆ ใน monolith
- **Field ที่เครื่องอ่านได้** structured logs สามารถถูก filter, aggregate และตั้ง alert ตาม field (`level`, `service`, `reason`) แทนการ match ข้อความที่เปราะบาง

สิ่งที่มันมีต้นทุนกับคุณ:

- **คุณต้อง propagate correlation id** สร้างมันที่ขอบ (gateway หรือบริการแรก) และส่งต่อในทุกการเรียกออก — มักผ่าน header — ไม่เช่นนั้น chain จะขาดที่ hop แรกที่ลืม
- **ปริมาณและต้นทุน** การรวมศูนย์ทุกบรรทัดจากทุก replica ผลิตข้อมูลจำนวนมาก คุณจะต้องมี log level ที่สมเหตุสมผล, sampling สำหรับ debug log ที่มีปริมาณสูง และนโยบาย retention ไม่เช่นนั้นทั้งบิลและเสียงรบกวนก็จะโตอย่างไม่มีขอบเขต
- **ระเบียบวินัยเรื่อง field** คุณค่ามาจากชื่อ field ที่ *สอดคล้องกัน* ข้ามบริการ หากไม่มี logging convention ที่ใช้ร่วมกัน `userId` ที่นี่และ `user_id` ที่นั่นก็จะทำลาย query ของคุณอย่างเงียบ ๆ

## Pattern ที่เกี่ยวข้อง

- [Distributed Tracing](/microservices-design-from-zero-to-hero/th/observability/distributed-tracing/) — trace id คือแนวคิดเดียวกันที่ถูกขยายต่อไป โดยเพิ่ม causal spans และ timing ทับลงบน correlation
- [Audit Logging](/microservices-design-from-zero-to-hero/th/observability/audit-logging/) — record แยกต่างหากที่คงทนสำหรับการกำกับดูแล แตกต่างจาก operational debug logs เหล่านี้
