## บริบท

service ของคุณคุยกันอยู่ตลอดเวลา และทุก call เหล่านั้นต้องการมาตรการป้องกันชุดเดียวกันไม่กี่อย่าง: traffic ควรถูกเข้ารหัส, request ที่ล้มเหลวควรถูก retry ตามจำนวนที่จำกัด, callee ที่ช้าควรชน timeout แทนที่จะค้าง caller ไว้ และทุก call ควรปล่อย metric และ trace span ออกมา ในโมดูล reliability และ observability คุณได้เรียนรู้การ build สิ่งเหล่านี้เข้าไปใน service — circuit breaker, retry, การ instrument

ปัญหาคือตอนนี้คุณมียี่สิบ service ในสี่ภาษา และแต่ละตัวต้อง implement มันทั้งหมดให้ถูกต้องและเหมือนกัน retry logic ของทีม Go ต่างจากของทีม Java เล็กน้อย mTLS ถูกตั้งค่าสามแบบต่างกัน service หนึ่งลืมปล่อย trace ไปเลย networking logic แบบ cross-cutting ตัวเดียวกันถูก implement ใหม่ ไม่สอดคล้องกัน อยู่ทุกที่

## ปัญหา

คุณต้องการให้ทุก call ระหว่าง service ถูกเข้ารหัส, retry, timeout, ถูกสังเกต และ route ตาม policy เดียวที่สอดคล้องกัน — โดยไม่ต้องขอให้ทุกทีมในทุกภาษาเขียนและดูแล logic นั้นด้วยมือ คุณต้องการเปลี่ยน policy (เข้มงวด timeout, เลื่อน 5% ของ traffic ไปยัง canary) จากส่วนกลางโดยไม่ต้อง redeploy service และคุณต้องการให้มันทำงานเหมือนกันไม่ว่า service จะเป็น Go, Java, Python หรือ Rust

ดังนั้นแรงที่ขัดแย้งกันคือ: พฤติกรรม networking ที่สอดคล้องกันและถูกควบคุมจากส่วนกลาง ข้ามกองทัพแบบ polyglot โดยใช้กับ service ที่แบก business logic โดยไม่ต้องแก้ไขหรือ redeploy มัน

## ทางออก

รับเอา **service mesh** มาใช้ mesh ย้ายเรื่อง communication เหล่านี้ทั้งหมดออกจาก service และเข้าไปไว้ใน **sidecar proxy** — process proxy เล็ก ๆ ที่ deploy ในหน่วยเดียวกับแต่ละ service instance ทุก byte ที่ service ส่งหรือรับจะผ่าน sidecar ของมัน sidecar ไม่ใช่ code ของคุณ ที่ทำ mutual TLS, ใช้ retry และ timeout, บังคับ routing rule และบันทึก metric และ trace

เพราะ proxy อยู่ในเส้นทาง network และเป็นกลางต่อภาษา service จึงไม่รับรู้ถึงมัน Go service และ Java service ได้พฤติกรรม mTLS, retry และ telemetry เหมือนกัน เพราะ proxy ตัวเดียวกัน implement มันให้ทั้งสอง

กองทัพของ sidecar คือ **data plane** — มันแบก traffic จริง ส่วน **control plane** ที่แยกต่างหากเป็นผู้ตั้งค่ามัน: คุณแสดง policy ครั้งเดียว (route นี้, timeout นี้, traffic split นี้) แล้ว control plane จะ push configuration ที่ได้ไปยังทุก sidecar การเปลี่ยน policy คือการ update control plane ไม่ใช่การ redeploy service

## ตัวอย่าง

คุณอธิบายพฤติกรรม traffic เป็น policy ที่ control plane กระจายออกไป policy นี้ตั้ง timeout และ retry แบบจำกัดสำหรับ call ไปยัง payment service และ split traffic เพื่อให้ 90% ไปยังเวอร์ชัน stable และ 10% ไปยัง canary — โดยไม่ต้องไปแตะ code ของ payment service เลย

```yaml
apiVersion: networking.mesh.io/v1
kind: TrafficPolicy
metadata:
  name: payment-routing
spec:
  host: payment-service
  timeout: 2s
  retries:
    attempts: 3
    perTryTimeout: 800ms
    retryOn: 5xx,connect-failure
  mtls:
    mode: STRICT
  trafficSplit:
    - version: stable
      weight: 90
    - version: canary
      weight: 10
```

control plane แปลสิ่งนี้เป็น configuration สำหรับทุก sidecar ที่ call `payment-service` เพื่อเลื่อน canary ไปข้างหน้า คุณเปลี่ยน weight ที่นี่ ไม่มี service ใดถูก build ใหม่หรือ redeploy

## บริบทที่ได้

สิ่งที่คุณได้:

- **networking แบบ cross-cutting ที่สอดคล้องกัน** mTLS, retry, timeout และ telemetry ทำงานเหมือนกันสำหรับทุก service ไม่ว่าจะภาษาใด เพราะ proxy ตัวเดียว implement มันทั้งหมด
- **การควบคุมจากส่วนกลางโดยไม่ต้องเขียน code** policy ด้าน routing, security และ resilience อยู่ใน control plane การปล่อย canary, การเลื่อน traffic และการ timeout ที่เข้มขึ้นเป็นการเปลี่ยน configuration ไม่ใช่การ redeploy
- **observability และ zero-trust ในตัว** ทุก hop ถูก trace และวัดอัตโนมัติ และ traffic ระหว่าง service ถูกเข้ารหัสและพิสูจน์ตัวตนโดยค่าเริ่มต้น

สิ่งที่คุณต้องเสีย:

- **latency และการใช้ resource ที่เพิ่มขึ้น** ทุก call ตอนนี้ผ่าน proxy สองตัว เพิ่มภาษี latency เล็กน้อยแต่มีจริง และเพิ่ม CPU และ memory เท่ากับ sidecar หนึ่งตัวต่อหนึ่ง service instance
- **ความซับซ้อนด้าน operation** mesh เป็นระบบ distributed ที่ซับซ้อนซึ่งคุณต้องติดตั้ง, upgrade และ debug control plane ที่ตั้งค่าผิดสามารถทำลาย communication ทั่วทั้งกองทัพได้
- **อีกชั้นหนึ่งที่ต้องคิดถึง** เมื่อ call ล้มเหลว คำตอบอาจอยู่ใน service ของคุณ, sidecar หรือ policy ของ control plane — มีชิ้นส่วนที่ขยับได้มากขึ้นระหว่างต้นเหตุกับอาการ

กฎที่ใช้ได้จริง: mesh คุ้มค่าเมื่อคุณมี service แบบ polyglot จำนวนมากและมีความต้องการจริงด้าน mTLS, การควบคุม traffic และ telemetry ที่สม่ำเสมอ สำหรับ service ไม่กี่ตัว มันมักเป็นเครื่องจักรที่มากเกินกว่าที่ปัญหาต้องการ

## Pattern ที่เกี่ยวข้อง

- [Service per Container](/microservices-design-from-zero-to-hero/th/deployment-and-cross-cutting/service-per-container/) — sidecar มักเป็น container ตัวที่สองที่ deploy ข้าง service container
- [Microservice Chassis](/microservices-design-from-zero-to-hero/th/deployment-and-cross-cutting/microservice-chassis/) — ทางเลือกแบบ in-process สำหรับบางเรื่องเดียวกัน mesh และ chassis มักถูกใช้ร่วมกัน
- [Externalized Configuration](/microservices-design-from-zero-to-hero/th/deployment-and-cross-cutting/externalized-configuration/) — policy ของ mesh เองก็เป็น externalized configuration ที่ถูก push โดย control plane

</Callout>
