## บริบท (Context)

คุณกำลังจะแกะระบบออกเป็น service และคุณต้องการหลักการแรกมาชี้นำการตัดแบ่ง คุณต้องการเส้นแบ่งที่ยังคงมีเหตุผลในอีกสามปีข้างหน้า มีทีมเป็นเจ้าของที่สามารถส่งมอบงานได้โดยไม่ต้องเจรจาข้ามทีมตลอดเวลา สัญชาตญาณของวิศวกรหลายคนคือการหั่นตามเทคโนโลยี — service หนึ่งสำหรับ API, หนึ่งสำหรับ workflow engine, หนึ่งสำหรับ reporting jobs — เพราะนั่นคือวิธีที่ codebase ถูกจัดเรียงอยู่แล้วในหัวของพวกเขา

## ปัญหา (Problem)

เทคโนโลยีเปลี่ยนเร็วกว่าสิ่งที่ธุรกิจทำจริงมาก framework มาแล้วก็ไป, data store ถูกสับเปลี่ยน, API ย้ายจาก REST ไป gRPC — แต่ผู้ค้าปลีกก็ยังรับ order, เก็บเงินลูกค้า, และจัดส่งสินค้าอยู่ดี ถ้าคุณวาดเส้นแบ่งของ service รอบ ๆ เทคโนโลยีของวันนี้ ทุกครั้งที่เทคโนโลยีเหล่านั้นเปลี่ยนก็จะบังคับให้คุณต้อง decompose ใหม่ ที่แย่กว่านั้นคือ การหั่นตามชั้นรับประกันได้เลยว่าการเปลี่ยนแปลงทางธุรกิจที่ง่ายที่สุด — เช่น เพิ่มกฎส่วนลด — จะไปแตะ web service, logic service, และ data service พร้อมกัน เส้นที่คุณวาดไม่ตรงกับเส้นที่ระบบเปลี่ยนแปลงจริง

ดังนั้นแรงต่าง ๆ คือ คุณต้องการเส้นแบ่งที่ *มั่นคง* ที่ให้ *cohesion สูง* (ทุกอย่างที่เปลี่ยนพร้อมกันอยู่ด้วยกัน) และ *coupling ต่ำ* (service แทบไม่ต้องเปลี่ยนพร้อม ๆ กัน) และที่แมปลงไปบนโครงสร้างขององค์กรได้อย่างหมดจด

## วิธีแก้ (Solution)

**business capability** คือสิ่งที่องค์กรทำเพื่อสร้างคุณค่า มันตอบคำถามว่า "ธุรกิจนี้ *ทำ* อะไร?" โดยจงใจมองข้ามว่า *ทำอย่างไร* ในปัจจุบัน capability มั่นคงก็เพราะมันถูกนิยามด้วยวัตถุประสงค์ ไม่ใช่ implementation สำหรับบริษัท e-commerce capability อาจรวมถึง Order Management, Product Catalog, Inventory, Billing, Shipping, และ Customer Accounts

pattern นี้คือการ **นิยามหนึ่ง service ต่อหนึ่ง business capability** (หรือต่อหนึ่งกลุ่ม capability ที่เกี่ยวข้องกันอย่างใกล้ชิด) คุณระบุ capability ได้ด้วยการศึกษาองค์กร คือ วัตถุประสงค์, โครงสร้าง, และ value stream ที่มันดำเนินการ capability มักเป็นลำดับชั้น — Order Management อาจประกอบด้วย Order Capture, Order Fulfilment, และ Returns — ซึ่งให้คันโยกตามธรรมชาติแก่คุณในการเลือกว่าจะทำให้แต่ละ service หยาบหรือละเอียดแค่ไหน

เพราะ capability สะท้อนว่าธุรกิจ *ทำอะไร* มากกว่า *ทำอย่างไร* เส้นแบ่งที่ได้จึงทนทาน คุณสามารถเขียนภายในของ service ใหม่ สับเปลี่ยน database หรือเปลี่ยน API ของมัน แล้วเส้นแบ่งก็ยังคงอยู่ เพราะ capability ที่มันเป็นตัวแทนนั้นไม่ได้เปลี่ยน

## ตัวอย่าง

## บริบทที่ได้ตามมา (Resulting context)

สิ่งที่คุณได้:

- **เส้นแบ่งที่มั่นคง** capability เปลี่ยนช้ากว่าเทคโนโลยีมาก service ที่วาดรอบ ๆ มันจึงแทบไม่ต้องถูกตัดใหม่ การเขียนใหม่ยังคงอยู่ภายใน service เดียว
- **cohesion สูง, coupling ต่ำ** ทุกอย่างที่จำเป็นต่อการทำ capability หนึ่งอยู่ในที่เดียว ดังนั้นการเปลี่ยนแปลงทางธุรกิจส่วนใหญ่จึงตกลงใน service เดียวแทนที่จะกระเพื่อมข้ามหลาย service
- **การแมปลงองค์กรอย่างหมดจด** capability มักเรียงตรงกับวิธีที่ธุรกิจคิดเกี่ยวกับตัวเองอยู่แล้ว ซึ่งทำให้การกำหนดความเป็นเจ้าของและความรับผิดชอบเป็นเรื่องง่าย

สิ่งที่คุณต้องจ่าย:

- **คุณต้องเข้าใจธุรกิจจริง ๆ** นี่คือส่วนที่ยาก การระบุ capability ได้ดีต้องอาศัยความรู้เรื่อง domain และการพูดคุยกับคนนอกฝ่ายวิศวกรรม map ที่ตื้นเขินจะให้เส้นแบ่งที่ตื้นเขิน
- **ความหยาบ-ละเอียดเป็นเรื่องของวิจารณญาณ** capability ซ้อนกันได้ การตัดสินว่า Order Capture และ Returns เป็นหนึ่ง service หรือสองนั้นไม่ใช่เรื่องกลไก หยาบเกินไปคุณก็สร้าง mini-monolith ขึ้นมาใหม่ ละเอียดเกินไปคุณก็จมอยู่กับการคุยกันข้าม service
- **มันไม่ได้ให้โมเดลร่วมแก่คุณด้วยตัวมันเอง** สอง service อาจใช้คำว่า "order" เพื่อหมายถึงสิ่งที่ต่างกันเล็กน้อย การ decompose ด้วย capability หยุดก่อนจะแก้ปัญหานั้น ซึ่งเป็นจุดที่การ decompose ด้วย subdomain เข้ามารับช่วงพอดี

## pattern ที่เกี่ยวข้อง

- [Decompose by Subdomain](/microservices-design-from-zero-to-hero/th/decomposition/decompose-by-subdomain/) — วิธีที่เสริมกันและขับเคลื่อนด้วยโมเดล ในการค้นหาและทำให้เส้นแบ่งเดียวกันคมชัดขึ้น
- [Service per Team](/microservices-design-from-zero-to-hero/th/decomposition/service-per-team/) — เพราะ capability แมปลงทีมได้ มันจึงทำให้ความเป็นเจ้าของเป็นเรื่องธรรมชาติ
- [ภาพรวมการ decompose](/microservices-design-from-zero-to-hero/th/decomposition/) — กลยุทธ์นี้เข้ากันกับกลยุทธ์อื่น ๆ อย่างไร
