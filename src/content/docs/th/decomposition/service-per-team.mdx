## บริบท (Context)

คุณมีชุด service ที่ดี วาดรอบ ๆ capability และ subdomain และออกแบบมาให้รันได้อย่างอิสระ จุดประสงค์ของงานทั้งหมดนั้นคือความเร็ว — ทีมส่งมอบงานแบบขนานโดยไม่ต้องรอกันและกัน แต่ความเป็นเจ้าของถูกปล่อยให้คลุมเครือ หลายทีมไปแตะ Order service, ทีม "platform" ร่วมเป็นทีมเดียวที่ได้รับอนุญาตให้ deploy Billing, และการเปลี่ยน Shipping ต้องได้รับการอนุมัติจากกลุ่มที่ไม่ได้ทำงานกับมันจริง ๆ สถาปัตยกรรมถูก decompose แล้ว แต่องค์กรรอบ ๆ มันไม่ได้ถูก decompose ด้วย และความเป็นอิสระที่สัญญาไว้ก็ไม่เคยมาถึง

## ปัญหา (Problem)

**Conway's law** สังเกตว่าโครงสร้างของระบบมักสะท้อนโครงสร้างการสื่อสารขององค์กรที่สร้างมัน นี่ไม่ใช่สโลแกนแต่เป็นข้อจำกัด ถ้าสามทีมสร้างหนึ่ง service service นั้นก็จะงอกรอยต่อเท่ามูลค่าของสามทีม และทุกการเปลี่ยนแปลงต้องการการเจรจาสามฝ่าย ถ้าทีมหนึ่งต้องไปแตะสี่ service เพื่อส่งมอบ feature service ทั้งสี่นั้นก็จะ couple เข้ากับตารางการ release ของทีมนั้นอย่างเงียบ ๆ ความเป็นเจ้าของร่วมเปลี่ยนทุกการ deploy ให้เป็นปัญหาการประสานงาน — ใครได้รับอนุญาตให้ merge, ใครต้อง review, การทดสอบของใครต้องผ่าน, ใคร on call เมื่อมันพัง ต้นทุนของการเปลี่ยนแปลงกลายเป็นถูกครอบงำ ไม่ใช่ด้วย code แต่ด้วยจำนวนคนที่ต้องเห็นพ้องกัน

ดังนั้นแรงต่าง ๆ คือ คุณต้องการให้แต่ละ service ถูกเปลี่ยนแปลงและ deploy ได้อย่างรวดเร็วและมั่นใจ แต่ความเป็นเจ้าของที่กระจายไปทั่วหลายทีม (หรือหลายทีมที่อัดแน่นอยู่บน service เดียว) นำการประสานงานข้ามทีมที่การ decompose ตั้งใจจะกำจัดกลับมาเสียพอดี

## วิธีแก้ (Solution)

**Service per Team** ทำให้ความเป็นเจ้าของชัดเจนและผูกขาด คือ **แต่ละ service ถูกเป็นเจ้าของโดยทีมเดียวเท่านั้น และทีมหนึ่งเป็นเจ้าของชุด service ที่เกี่ยวข้องกันขนาดเล็กแบบ end to end** — ออกแบบ, เขียน code, ทดสอบ, deploy, และดำเนินการ ไม่มีทีมอื่น deploy service ของคุณ ไม่มี service ใดมีสองนาย ที่สำคัญคือ คุณไม่ปล่อยให้ผังองค์กรหลุดออกมาโดยบังเอิญ — คุณปฏิบัติต่อ Conway's law เป็นเครื่องมือออกแบบ (*inverse Conway manoeuvre*) และปั้นทีมอย่างจงใจให้เข้ากับเส้นแบ่งของ service ที่คุณต้องการ

ทีมหนึ่งถูกกำหนดขนาดให้เป็นเจ้าของ service ของมันได้อย่างสบาย — เล็กพอที่จะประสานงานกันภายในตัวเอง ใหญ่พอที่จะแบกภาระ on-call และการส่งมอบ ทีมทำงานร่วมกันแบบเดียวกับที่ service ทำ คือ ผ่าน API และ contract ที่มั่นคงและเผยแพร่แล้ว แทนที่จะผ่าน code ร่วมหรือ deployment pipeline ร่วม เมื่อทีม A ต้องการบางอย่างจาก service ของทีม B มันก็ขอผ่าน interface ที่มีเวอร์ชัน ไม่ใช่ด้วยการไปแก้ code ของ B

ผลลัพธ์คือองค์กรและสถาปัตยกรรมเสริมแรงซึ่งกันและกัน เส้นแบ่งที่สอดคล้องกันหมายความว่าการเปลี่ยนแปลงส่วนใหญ่อยู่ภายใน service ของทีมเดียว ดังนั้นทีมนั้นจึงสามารถตัดสินใจ, สร้าง, ทดสอบ, และ release ตามจังหวะของตัวเองได้ — ซึ่งเป็นความเป็นอิสระที่ความพยายามทั้งหมดนี้มีไว้เพื่อสิ่งนั้น

## บริบทที่ได้ตามมา (Resulting context)

สิ่งที่คุณได้:

- **ความเป็นอิสระและความเร็วที่แท้จริง** ทีมที่เป็นเจ้าของ service ของตนแบบ end to end สามารถออกแบบ, สร้าง, ทดสอบ, และ deploy ได้โดยไม่ต้องรอใคร ดังนั้นการเปลี่ยนแปลงส่วนใหญ่จึงส่งมอบได้ตามจังหวะของทีมเอง
- **ความรับผิดชอบที่ชัดเจน** มีทีมเดียวเท่านั้นที่เป็นเจ้าของแต่ละ service ซึ่งขจัดความกำกวมเรื่อง "ใครรับผิดชอบ?" — สำหรับการ review code, สำหรับ deploy pipeline, และสำหรับเครื่องเรียกเมื่อมันพัง
- **สถาปัตยกรรมที่เข้ากับองค์กร** ด้วยการปั้นทีมให้เข้ากับเส้นแบ่งของ service ที่ต้องการ คุณหยุดต่อสู้กับ Conway's law และเริ่มใช้มัน รอยต่อในระบบลงเอยตรงที่รอยต่อระหว่างทีมอยู่แล้ว

สิ่งที่คุณต้องจ่าย:

- **การประสานงานถูกผลักไปที่เส้นแบ่ง** ความเป็นอิสระภายในทีมมาพร้อมราคาของ contract ที่มีวินัยระหว่างทีม การเปลี่ยนแปลงที่ข้ามเส้นแบ่งของ service ก็ยังต้องการความเห็นพ้องข้ามทีม ดังนั้นเส้นแบ่งที่วางไว้ไม่ดีก็เพียงย้ายความเจ็บปวดของการประสานงานแทนที่จะกำจัดมัน
- **การปรับองค์กรเป็นเรื่องยากและการเมือง** การจัดทีมให้เข้ากับ service อาจหมายถึงการเปลี่ยนสายการรายงานและจำนวนคน — การเปลี่ยนแปลงเชิงองค์กรที่ช้ากว่าและละเอียดอ่อนกว่าการเปลี่ยน code ใด ๆ
- **ความเสี่ยงของ coupling และความซ้ำซ้อน** หากไม่ระวัง ทีมก็จะ couple ผ่าน library และ database ร่วม (สร้าง distributed monolith ขึ้นมาใหม่) หรือทำงานซ้ำซ้อนเพราะไม่มีใครเป็นเจ้าของพื้นที่ส่วนกลาง ทั้งสองอย่างต้องการการกำกับดูแลอย่างจริงจังเพื่อหลีกเลี่ยง

## pattern ที่เกี่ยวข้อง

- [Decompose by Business Capability](/microservices-design-from-zero-to-hero/th/decomposition/decompose-by-business-capability/) — capability แมปลงทีมได้อย่างหมดจด ทำให้การกำหนดความเป็นเจ้าของง่าย
- [Decompose by Subdomain](/microservices-design-from-zero-to-hero/th/decomposition/decompose-by-subdomain/) — bounded context ให้แต่ละทีมมีส่วนที่ self-contained ไว้เป็นเจ้าของ
- [Self-Contained Service](/microservices-design-from-zero-to-hero/th/decomposition/self-contained-service/) — ความเป็นอิสระตอน runtime และความเป็นอิสระของทีมเสริมแรงซึ่งกันและกัน ทีมที่ไม่ขึ้นกับใครตอน deploy ก็ต้องการ service ที่ไม่ขึ้นกับใครตอน runtime เช่นกัน

<Callout title="inverse Conway manoeuvre">
Conway's law มักถูกยกมาเป็นคำเตือน แต่มันก็เป็นคันโยกด้วย ถ้าสถาปัตยกรรมของคุณจะลงเอยดูเหมือนผังองค์กรของคุณอยู่ดี งั้นก็ *เลือกสถาปัตยกรรมที่คุณต้องการก่อนแล้วปั้นทีมให้เข้ากับมัน* การวาดเส้นแบ่งของทีมอย่างจงใจให้สะท้อนเส้นแบ่งของ service ที่คุณตั้งใจไว้ — แทนที่จะดัดแปลง service มาใส่ทีมที่มีอยู่แล้ว — คือวิธีที่คุณทำให้องค์กรผลักดันการออกแบบไปในทิศทางที่คุณต้องการ แทนที่จะผลักออกไปจากมัน
</Callout>
