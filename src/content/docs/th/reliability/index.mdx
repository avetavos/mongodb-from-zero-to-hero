## บริบท

ในระบบ monolith ฟังก์ชันหนึ่งเรียกอีกฟังก์ชันหนึ่ง การเรียกนั้นไม่ return ก็ throw และมัน return กลับมาในระดับนาโนวินาที ไม่มีเครือข่ายคั่นกลาง ไม่มี process แยกที่อาจกำลัง restart ไม่มีคิวที่อาจเต็ม เมื่อคุณแยกระบบออกเป็นหลายบริการ ทุก ๆ การเรียกภายใน process ที่ข้ามขอบเขตของบริการจะกลายเป็นการเรียกแบบ remote — ผ่าน socket ผ่าน load balancer ข้าม availability zone dependency ที่คุณเคยเชื่อถือได้อย่างไม่มีเงื่อนไข ตอนนี้กลายเป็นโปรแกรมแยกต่างหากที่อาจช้า อาจรับภาระเกิน หรืออาจหายไปเฉย ๆ

## ปัญหา

ข้อเท็จจริงที่นิยามระบบแบบกระจายคือ **partial failure**: ในทุก ๆ ขณะ มีบางส่วนของระบบที่พังในขณะที่ส่วนที่เหลือยังทำงานต่อไป dependency หนึ่งไม่ได้มีแค่สถานะ "ขึ้น" หรือ "ลง" — มันสามารถ *ช้า* ได้ ซึ่งแย่กว่า เพราะ dependency ที่ช้าจะจับทรัพยากรของคุณไว้เป็นตัวประกัน ลองนึกถึง order service ที่เรียก payment service ในทุก ๆ request payment service เริ่มตอบสนองในเวลาแปดวินาทีแทนที่จะเป็นแปดสิบมิลลิวินาที ตอนนี้ order request แต่ละครั้งจะจอด thread ทิ้งไว้เพื่อรอ payment thread กองพะเนินขึ้นเรื่อย ๆ connection pool ระบายจนหมด และภายในไม่กี่วินาที order service ก็หยุดตอบ health check — ทั้ง ๆ ที่ *ตัว order service เองไม่ได้พังอะไรเลย* ความล้มเหลวได้แพร่ขึ้นไปต้นทาง

นี่คือ **cascading failure**: คอมโพเนนต์หนึ่งที่กำลังดิ้นรนดึงผู้เรียกของมันลงไปด้วย ซึ่งไปดึงผู้เรียกของผู้เรียกอีกที จนกระทั่ง dependency ที่ช้าตัวเดียวล้มทั้งสายของบริการที่แข็งแรงดี

ดังนั้นคำถามหลักของโมดูลนี้คือ: คุณจะหยุด dependency ที่ช้าหรือล้มเหลวไม่ให้กินทรัพยากรของทุกสิ่งที่พึ่งพามันได้อย่างไร เพื่อให้ความล้มเหลวหนึ่งจุดถูกควบคุมไว้แทนที่จะแพร่กระจาย

## วิธีแก้

คุณไม่สามารถป้องกันไม่ให้ dependency ล้มเหลวได้ สิ่งที่คุณ *ทำได้* คือตัดสินใจล่วงหน้าว่าบริการของคุณจะมีพฤติกรรมอย่างไรเมื่อมันล้มเหลว — และทำให้พฤติกรรมนั้นมีขอบเขต รวดเร็ว และแยกตัวออกมา pattern ต่าง ๆ ในโมดูลนี้คือชิ้นส่วนที่ใช้สร้างพฤติกรรมนั้น ไม่มีอันไหนแปลกประหลาด เมื่อรวมกันแล้วมันเปลี่ยน "dependency ช้าแล้วเราล้มตาม" ให้กลายเป็น "dependency ช้าแล้วเราเสื่อมประสิทธิภาพลงอย่างนุ่มนวล"

- **Circuit Breaker** — เมื่อการเรียก dependency ข้ามเกณฑ์ error ก็หยุดเรียกมันไปสักพัก fail fast แทนที่จะกอง request ที่ไร้ความหวังไว้ และตรวจวัดเป็นระยะเพื่อตรวจจับการฟื้นตัว
- **Retry + Timeout** — กำหนดขอบเขตเวลาที่เข้มงวดให้กับทุกการเรียก remote เพื่อให้มันค้างตลอดกาลไม่ได้ และ retry ความล้มเหลวแบบ *transient* ด้วย exponential backoff และ jitter โดยเล็งไปที่ idempotent operation เท่านั้น
- **Bulkhead** — แยกทรัพยากร (thread pool, connection pool, semaphore) ที่ใช้สำหรับแต่ละ dependency เพื่อให้ dependency หนึ่งที่อิ่มตัวไม่สามารถระบายทรัพยากรที่ส่วนที่เหลือของระบบต้องการได้
- **Rate Limiting / Throttling** — จำกัดอัตราของงานที่เข้ามา เพื่อไม่ให้ traffic ที่พุ่งสูงหรือ client ที่มีพฤติกรรมผิดปกติดันบริการให้เกินกำลังที่มันรองรับได้จริง โดยทิ้งหรือเข้าคิวส่วนที่เกิน

pattern เหล่านี้เสริมซึ่งกันและกัน timeout ทำให้การนับ error ของ circuit breaker มีความหมาย circuit breaker กันไม่ให้ retry กลายเป็น retry storm bulkhead ควบคุมสิ่งที่สองอย่างแรกพลาดไป และ rate limiting ป้องกันไม่ให้บริการถูกถาโถมตั้งแต่แรก

## โมดูลนี้ครอบคลุมอะไรบ้าง

เราเริ่มต้นด้วย **circuit breaker** ซึ่งเป็น pattern ที่เปลี่ยน dependency ที่ค้างให้กลายเป็นความล้มเหลวที่รวดเร็วและคาดเดาได้ จากนั้นเราเพิ่ม **retry และ timeout** ซึ่งเป็นวินัยระดับการเรียกแต่ละครั้งที่จำกัด latency และฟื้นตัวจากการสะดุดชั่วครู่ ต่อมาคือ **bulkhead** ซึ่งควบคุมความเสียหายด้วยการแบ่งกั้นทรัพยากร เราปิดท้ายด้วย **rate limiting** ซึ่งปกป้องกำลังของบริการจากการรับภาระเกิน แต่ละบทเรียนมีรูปแบบเดียวกัน: สถานการณ์ที่นำไปสู่ pattern แรงที่ตึงเครียดกัน สิ่งที่ pattern ทำ ไดอะแกรม โค้ดตัวอย่างที่รันได้ในสี่ภาษา และการมองอย่างตรงไปตรงมาว่า pattern นั้นมีต้นทุนอะไรกับคุณ

## Pattern ที่เกี่ยวข้อง

- [Circuit Breaker](/microservices-design-from-zero-to-hero/th/reliability/circuit-breaker/) — เริ่มที่นี่ มันคือ pattern ที่หยุดการค้างไม่ให้กลายเป็นการล่ม
- [Retry and Timeout](/microservices-design-from-zero-to-hero/th/reliability/retry-and-timeout/) — วินัยระดับการเรียกแต่ละครั้งที่ breaker สร้างขึ้นอยู่บนนั้น
