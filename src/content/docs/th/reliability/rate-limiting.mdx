## บริบท

ทุกบริการมีเพดาน instance หนึ่งสามารถรับ request ได้เพียงจำนวนหนึ่งต่อวินาทีก่อนที่ CPU จะอิ่มตัว connection pool เต็ม หรือหน่วยความจำพุ่งสูง ส่วนใหญ่คุณทำงานอยู่ต่ำกว่าเพดานนั้นอย่างสบาย ๆ แต่ traffic ไม่ราบเรียบ: อีเมลการตลาดถูกส่งออกไป client ยอดนิยม deploy retry loop ที่แน่นเกินไป scraper ค้นพบ API ของคุณ หรือเหตุล่มที่ต้นทางจู่ ๆ เปลี่ยนทิศทาง request จำนวนมหาศาลมาทางคุณ

pattern ก่อนหน้าในโมดูลนี้ปกป้องคุณจากการที่ *dependency ของคุณ* ล้มเหลว rate limiting ปกป้องคุณจาก *ผู้เรียกของคุณ* — และจากความสำเร็จของคุณเอง

## ปัญหา

เมื่อ demand เกินกำลัง บริการไม่ได้ล้มเหลวอย่างนุ่มนวลด้วยตัวเอง มันคว่ำ เมื่อ request มาถึงเร็วกว่าที่ทำเสร็จ คิวก็โต latency พุ่ง timeout ทำงาน และ client retry — เพิ่มภาระ *มากขึ้น* ให้บริการที่รับภาระเกินอยู่แล้ว เลยจุดพลิกผัน throughput ไม่ได้แค่หยุดนิ่ง แต่มันพังทลายลง เพราะเซิร์ฟเวอร์ใช้เวลาไปกับการจัดการ backlog ที่มันไม่มีวันเคลียร์ได้ บริการที่รับ request ได้สบาย ๆ 1,000 ครั้งต่อวินาทีอาจกลายเป็นรับได้ *ศูนย์* ภายใต้น้ำท่วม 5,000 ครั้ง เพราะทุก request timeout ก่อนที่จะเสร็จ

ดังนั้นแรงที่ตึงเครียดกันคือ: คุณมีกำลังที่จำกัด demand อาจพุ่งสูงเกินกว่านั้นอย่างคาดเดาไม่ได้ และการให้บริการ *ทุกอย่างได้แย่* แย่กว่าการให้บริการ *สิ่งส่วนใหญ่ได้ดี* — คุณต้องรักษาให้บริการทำงานอยู่ภายในกรอบที่ปลอดภัยแม้เมื่อมีงานเข้ามามากกว่าที่มันรับไหว

## วิธีแก้

**rate limiting** จำกัดว่าบริการจะรับงานได้มากเพียงใดตามเวลา รักษาให้มันอยู่ภายในกรอบที่ปลอดภัย เมื่อ request มาถึงเร็วกว่าขีดจำกัดที่กำหนด ส่วนเกินจะถูก **shed** (ปฏิเสธทันที โดยปกติด้วย HTTP `429 Too Many Requests`) หรือ **เข้าคิว** ชั่วครู่เพื่อปรับช่วง burst สั้น ๆ ให้ราบรื่น ไม่ว่าจะแบบไหน งานที่ *ได้รับ* เข้ามาก็จะได้รับการให้บริการที่ดี

กลไกที่พบบ่อยที่สุดคือ **token bucket** ถัง (bucket) เก็บ token ได้สูงสุด `capacity` token และเติม (refill) ในอัตราคงที่ `refillPerSecond` token ทุก request ต้องหยิบ token หนึ่งตัวเพื่อดำเนินการ หากถังว่าง request จะถูกปฏิเสธ เพราะถังสามารถเก็บ token สำรองได้ มันจึงอนุญาตให้เกิด **burst** สั้น ๆ (ใช้ token ที่สะสมไว้) ในขณะที่ยังบังคับอัตราเฉลี่ยในระยะยาว (อัตรา refill) การผสานนั้น — burst ในระดับเล็ก มีขอบเขตในระดับใหญ่ — สอดคล้องกับพฤติกรรมจริงของ traffic

## ตัวอย่าง

นี่คือ limiter แบบ token-bucket มัน refill แบบ lazy ตามเวลาที่ผ่านไป จากนั้นจึงรับ request ก็ต่อเมื่อมี token พร้อมใช้ แต่ละตัวอย่างเป็นอิสระในตัวเองและเขียนตามแบบฉบับของแต่ละภาษา

  <TabItem label="TypeScript">
    ```ts
    class TokenBucket {
      private tokens: number;
      private last: number = Date.now();

      constructor(
        private readonly capacity: number,
        private readonly refillPerSec: number,
      ) {
        this.tokens = capacity;
      }

      private refill(): void {
        const now = Date.now();
        const elapsed = (now - this.last) / 1000;
        this.tokens = Math.min(this.capacity, this.tokens + elapsed * this.refillPerSec);
        this.last = now;
      }

      tryAcquire(): boolean {
        this.refill();
        if (this.tokens >= 1) {
          this.tokens -= 1;
          return true;
        }
        return false;
      }
    }

    const limiter = new TokenBucket(100, 50); // burst 100, sustain 50/s
    if (!limiter.tryAcquire()) {
      return new Response('Too Many Requests', { status: 429 });
    }
    ```

    ```python
    class TokenBucket:
        def __init__(self, capacity: int, refill_per_sec: float):
            self.capacity = capacity
            self.refill_per_sec = refill_per_sec
            self.tokens = float(capacity)

            self.last = now
    ```

    ```rust
    use std::time::Instant;

    pub struct TokenBucket {
        capacity: f64,
        refill_per_sec: f64,
        tokens: f64,
        last: Instant,
    }

    impl TokenBucket {
        pub fn new(capacity: f64, refill_per_sec: f64) -> Self {
            Self {
                capacity,
                refill_per_sec,
                tokens: capacity,
                last: Instant::now(),
            }
        }

        pub fn try_acquire(&mut self) -> bool {
            let now = Instant::now();
            let elapsed = now.duration_since(self.last).as_secs_f64();
            self.tokens = (self.tokens + elapsed * self.refill_per_sec).min(self.capacity);
            self.last = now;
            if self.tokens >= 1.0 {
                self.tokens -= 1.0;
                true
            } else {
                false
            }
    ```

## บริบทที่เกิดขึ้นตามมา

สิ่งที่คุณได้รับ:

- **กำลังที่ถูกปกป้อง** บริการจะไม่รับงานมากกว่าที่มันให้บริการได้ ดังนั้นมันจึงเสื่อมประสิทธิภาพด้วยการ *ปฏิเสธบาง* request แทนที่จะ *ล้มเหลวทั้งหมด* throughput จึงอยู่บนที่ราบแทนที่จะพังทลายเลยจุดพลิกผัน
- **ความเป็นธรรมและการควบคุมการใช้งานในทางที่ผิด** การจำกัดต่อ client (ต่อ API key, ต่อ IP) หยุดผู้เรียกที่เสียงดังหรือมุ่งร้ายตัวหนึ่งไม่ให้ทำให้คนอื่นทั้งหมดอดอยาก และให้คันโยกแก่คุณในการรับมือ retry loop ที่ควบคุมไม่ได้และ scraper
- **ความทนทานต่อ burst** token bucket ดูดซับการพุ่งสูงสั้น ๆ ที่ถูกต้องตามกฎหมายโดยใช้ capacity ที่เก็บไว้ ในขณะที่ยังกำหนดขอบเขตอัตราที่ยั่งยืน — คุณไม่ต้องเลือกระหว่าง "ราบเรียบเท่านั้น" กับ "ไม่มีขีดจำกัด"

สิ่งที่มันมีต้นทุนกับคุณ:

- **การเลือกขีดจำกัด** ตั้งอัตราต่ำเกินไปแล้วคุณจะปฏิเสธ traffic ที่บริการน่าจะรับได้ สูงเกินไป limiter ก็ไม่ปกป้องคุณ ตัวเลขมาจาก load test กับกำลังจริง ไม่ใช่การเดา และมันเปลี่ยนแปลงไปเมื่อบริการเปลี่ยนไป
- **การ return สัญญาณที่ถูกต้อง** request ที่ถูกปฏิเสธควร return `429 Too Many Requests` ในอุดมคติพร้อม header `Retry-After` เพื่อให้ client ที่มีพฤติกรรมดีถอยกลับแทนที่จะกระหน่ำ error เปล่า ๆ เป็นการเชื้อเชิญให้ retry ทันทีซึ่งทำลายล้าง limiter
- **การประสานงานแบบกระจาย (distributed coordination)** limiter ใน instance หนึ่งเห็นเฉพาะ traffic ของ instance นั้น การบังคับขีดจำกัดแบบ global ข้ามหลาย instance ต้องใช้สถานะที่แชร์กัน (central store เช่น Redis) ซึ่งเพิ่ม dependency และ latency บ้างใน hot path

## Pattern ที่เกี่ยวข้อง

- [Circuit Breaker](/microservices-design-from-zero-to-hero/th/reliability/circuit-breaker/) — ปกป้อง *คุณ* จาก dependency ที่ล้มเหลว ส่วน rate limiting ปกป้องบริการจาก *ผู้เรียกของมัน*
- [Bulkhead](/microservices-design-from-zero-to-hero/th/reliability/bulkhead/) — จำกัดความ concurrency ในขณะที่ rate limiting จำกัด throughput ทั้งสองประกอบกันได้ดี
- [Retry and Timeout](/microservices-design-from-zero-to-hero/th/reliability/retry-and-timeout/) — `429` พร้อม `Retry-After` บอกตรรกะการ retry ของ client อย่างชัดเจนว่าควรถอยกลับนานเท่าใด

<Callout title="การ shed load เป็นฟีเจอร์ ไม่ใช่ความล้มเหลว">
การปฏิเสธ request รู้สึกเหมือนการยอมแพ้ แต่ที่ขอบของกำลัง มันคือทางเลือกที่รับผิดชอบ บริการที่รับทุกอย่างแล้ว timeout กับทุกอย่างให้บริการ *ใครไม่ได้เลย* บริการที่ shed ส่วนล้นและ return `429` ที่ชัดเจนยังคงให้บริการคนส่วนใหญ่ได้ดีและบอกที่เหลือให้กลับมาในไม่ช้า การตัดสินใจว่าจะ shed *อะไร* ก่อน — traffic ที่มีลำดับความสำคัญต่ำ ผู้ใช้ที่ไม่ระบุตัวตน client ที่เสียงดังที่สุด — เปลี่ยนการ shed load จากเครื่องมือทื่อ ๆ ให้กลายเป็นกลยุทธ์การเสื่อมประสิทธิภาพที่จงใจ
</Callout>
